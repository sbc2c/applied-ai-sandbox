Feature I picked: Add a "starred" field to notes and a filter to show only starred notes.

## Plan from chat
Data model — add one field: "starred": false (boolean, defaults to false). No migration needed since it's in-memory; just update your factory/init function.
Route changes — minimal additions only:

PATCH /notes/:id/star — toggles starred on a single note, returns the updated note. Keeps starring as its own atomic action rather than going through a full PUT /notes/:id (avoids accidentally clobbering a concurrent title/body edit).
GET /notes?filter=starred — optional query param; backend filters before returning. If you already filter client-side this can be skipped entirely.

Edge cases to handle:

Starring the last starred note while in Starred view — the list goes empty. Show the empty state ("No starred notes yet") rather than collapsing to nothing.
Filter survives note edits — if you update a note's title/body, preserve the current filter so the user isn't bounced back to "All."
New notes default to starred: false — make sure your note creation function/factory explicitly sets this rather than leaving the field absent, so downstream filter logic doesn't need to handle undefined.
Optimistic UI vs. server error — if the PATCH call fails, toggle starred back and show an inline error near the note rather than a modal/toast, to keep it low-friction.
No notes at all vs. no starred notes — two distinct empty states. "You have no notes" vs. "No starred notes yet — star one from the All view."
Deep linking to ?filter=starred — if you ever add URL state, make sure the toggle reflects it on load.
